# 连接组与多规则转发验收记录

## 范围

- Issue：#18
- 开发分支：`codex/issue-18-connection-groups`
- 应用版本：`0.4.0` 开发测试包，不是公开 Release

## 自动化验证

- `swift test`：195 项测试通过。
- 混合本地、远程、动态 SOCKS 规则生成一个 `/usr/bin/ssh -N` 参数数组。
- 禁用规则不进入 SSH 参数；任意规则中的 Host 注入、非法端口或非法模式会阻止整个连接组启动。
- 组内和跨组本地监听冲突按通配地址重叠规则检查。
- 旧单端口 `tunnels.json` 解码为单规则连接组并保留 UUID、名称、标签、收藏、顺序、自动重连和自动启动。
- 首次写回新模型前创建 `tunnels.json.pre-connection-groups.bak`，权限为 `0600`，内容与旧文件原始字节一致。
- JSON 导出使用 schema v2；导入继续读取 schema v1，并清除风险确认、自动启动和最近使用时间。
- 连接组规则在编辑草稿中保持 UUID、模式、启用状态、URL 和顺序。
- 新建时区分连接组与 SSH Config 引用；已保存配置不允许在编辑页直接改变类型。
- 第一条和后续规则使用一致的规则卡片，并以“启用此规则”明确开关对象。
- 所有规则均停用时允许保存，但启动入口不可用且显示明确原因。
- 当前规则必填项未完成时不能继续添加空规则；填写完整后才可新增下一条，每个连接组最多 20 条规则。
- 本地端口状态刷新对全部规则共用一次 `lsof` 快照；连接建立阶段保持 2 秒检查，确认稳定监听后降为 30 秒；相同结果不再发布运行时字典，避免规则数量放大子进程和 SwiftUI 重绘开销。
- 持久化配置与单个 SSH Config 文件在读取前后执行 1 MiB 上限检查；配置中的打开地址在交给系统前再次限制为带 Host 的 HTTP/HTTPS URL。
- SSH 启动和 `ssh -G` 均使用固定可执行文件及参数数组，没有 shell 字符串拼接；Host、端口、危险监听、导入大小、配置数量、文件权限和诊断脱敏边界均已复核。

## 正式手工验收

1. 覆盖安装测试包后确认旧配置全部显示为单规则连接组，名称、标签、收藏、顺序和自动化设置不变。
2. 新建时分别检查“连接组”和“SSH Config 引用”入口；保存后重新编辑，确认不能直接改变配置类型。
3. 编辑一个应用内连接组，增加本地、远程和 SOCKS 规则，确认第一条和新增规则都显示“启用此规则”，并分别设置启用状态和 URL。
4. 停用其中一条规则后启动连接组，确认系统中只新增一个受应用管理的 SSH 进程，只有已启用规则对应的监听可用。
5. 停用全部规则，确认配置可以保存，但启动入口不可用并提示至少启用一条转发规则。
6. 再次启用全部规则并启动，确认三个监听均可用；停止连接组后确认全部规则同时停止且没有遗留 SSH 进程。
7. 在运行中编辑规则，确认界面先提示将重启整个连接组，保存后只有一个新 SSH 进程。
8. 制造组内和跨组本地端口冲突，确认启动前阻止并显示对应监听端口。
9. 修改危险规则的监听地址或端口，确认原风险确认失效；重新确认后可启动。
10. 创建 SSH Config 引用，确认只显示 Host 别名相关字段，不显示规则模式或“启用此规则”。
11. 导出配置，确认 `schemaVersion` 为 `2` 且规则位于 `rules` 数组；导入 v1 文件预览为单规则组且不会启动 SSH。
12. 确认 `tunnels.json.pre-connection-groups.bak` 存在、权限为 `0600`，可在退出应用后用于恢复旧结构。

## 回滚

退出应用，把 `tunnels.json.pre-connection-groups.bak` 复制回 `tunnels.json`，再安装旧版本应用。不得在应用运行中替换配置文件。
